Backend Architecture · Agentic AI · MCP · RAG

End-to-end design for a trustworthy career roadmap generator.

This architecture defines the backend system that generates personalised, realistic career roadmaps from user goals, CV evidence, market signals, learning resources, and progress history. It is designed to avoid made-up advice by requiring structured evidence, explicit uncertainty, human approval, source-aware retrieval, and automated validation before any roadmap becomes active.

Primary backendFastAPI, Python 3.12, Pydantic v2, SQLAlchemy
Agent runtimePlanner, RAG, market, validator, roadmap, coach agents
Knowledge layerPostgreSQL, pgvector/vector DB, object store, Redis
Tool layerMCP gateway with scoped, auditable external connectors

1. Design objective

The system must generate a living career roadmap for any user, not only Roger's AI Systems Engineer case. Roger's roadmap becomes a seed template and validation example, while the production system uses dynamic user discovery, CV analysis, market intelligence, and continuous tracking to produce a tailored plan.

01
Ask before assuming when critical user information is missing.
02
Ground every important recommendation in user evidence, retrieved knowledge, or live market data.
03
Generate structured roadmap artifacts: phases, milestones, weekly plan, resources, evidence targets, metrics.
04
Continuously adapt the roadmap as the user progresses and the market changes.

3. Architecture principles

These principles prevent the AI layer from becoming a black-box chatbot and keep the output practical, reviewable, and safe.

Evidence-first generation

The LLM may reason and compose, but it cannot invent market facts, salary ranges, role requirements, certifications, or company-specific claims without evidence records.

Clarify before planning

If target role, country, timeline, weekly availability, skill level, constraints, or CV evidence are missing, the system produces clarification questions instead of a final roadmap.

Human-approved actions

Roadmap updates, outreach drafts, CV rewrites, calendar writes, and external account actions require user review and explicit approval unless the action is read-only.

Agent separation

One agent should not collect evidence, generate a plan, and validate itself. Specialist agents communicate through typed messages and audited shared state.

Structured outputs only

Roadmaps, gap analyses, tasks, milestones, and resource recommendations are generated as JSON matching Pydantic schemas before being rendered in the UI.

Continuous adaptation

The plan is not a one-time artifact. Weekly reviews, completed milestones, new CVs, job-market changes, and user preference changes trigger re-evaluation.

4. Agentic orchestration backend

The orchestration service is the control plane. It receives user intent, creates a roadmap generation job, decides which agents are needed, enforces policies, streams progress back to the client, and persists only validated outputs.

Core services

  • AgentOrchestratorService creates and supervises generation runs.
  • AgentMemoryService stores run state, user profile snapshots, and agent outputs.
  • ToolPolicyService decides which MCP tools an agent may call for the current user.
  • EvidenceService stores retrieved evidence, citations, freshness, and source quality.
  • RoadmapMaterializerService converts an approved roadmap into phases, tasks, habits, metrics, and reminders.

Run states

createdintake_startedneeds_clarificationretrieving_evidencegeneratingvalidatingawaiting_user_approvalmaterialized

Every state transition is written to an immutable audit log with user ID, run ID, triggering agent, input hash, output hash, token usage, tool calls, and validation result.

Orchestration pseudocode

async def generate_career_roadmap(user_id: UUID, request: RoadmapGenerationRequest):
    run = await runs.create(user_id=user_id, request=request)

    profile = await profile_agent.extract_profile(user_id, request)
    completeness = await clarification_agent.score(profile, request)

    if completeness.requires_clarification:
        questions = await clarification_agent.generate_questions(completeness.missing_slots)
        return RoadmapRunResult(status="needs_clarification", questions=questions)

    evidence_pack = await rag_agent.retrieve_context(profile, request)
    market_pack = await market_agent.fetch_market_signals(profile, request)
    gap_report = await gap_agent.compare(profile, request.target, evidence_pack, market_pack)

    draft = await roadmap_agent.synthesize(profile, gap_report, evidence_pack, market_pack)
    validation = await validator_agent.validate(draft, evidence_pack, market_pack)

    if not validation.passed:
        repaired = await roadmap_agent.repair(draft, validation.issues)
        validation = await validator_agent.validate(repaired, evidence_pack, market_pack)

    return RoadmapRunResult(
        status="awaiting_user_approval",
        roadmap=draft,
        evidence=evidence_pack.public_summary,
        validation=validation.public_summary,
    )

6. End-to-end roadmap generation lifecycle

A roadmap generation run should be deterministic at the workflow level, even though LLM reasoning is probabilistic. The workflow below defines the minimum reliable path.

STEP 01
IntakeUser enters goals, uploads CV, selects target role, country, timeline, weekly availability, constraints, learning preferences.
STEP 02
Clarification gateSystem checks required slots and asks follow-up questions instead of guessing when data is missing.
STEP 03
Evidence retrievalRAG agent retrieves user profile, prior plans, internal career frameworks, job-market context, and learning-resource metadata.
STEP 04
Gap analysisProfile agent and market agent compare current evidence against target-role requirements.
STEP 05
Roadmap synthesisRoadmap agent creates phases, milestones, projects, weekly plan, resources, metrics, and risk flags.
STEP 06
Validation & approvalCritic agent verifies evidence, realism, constraints, and uncertainty. User approves before materialization.

7. Clarification question engine

The system should not generate a final roadmap when it lacks essential inputs. It should ask the smallest number of high-value questions needed to proceed safely.

Required slots

  • Current role or background
  • Target role, industry, or exploration mode
  • Country or target market
  • Timeline and urgency
  • Weekly time budget
  • Skill level and CV evidence
  • Constraints: work, study, budget, visa, family

Question policy

  • Ask at most 3–5 questions per turn.
  • Prioritise blockers before preferences.
  • Offer multiple-choice answers when useful.
  • Explain why a question matters.
  • Allow “use a reasonable default” for non-critical fields.

Proceed conditions

  • All critical slots present.
  • Completeness score ≥ 0.82.
  • No high-risk contradiction in user profile.
  • At least one target or exploration objective exists.
  • Timeline and workload are feasible enough for a draft.

Example clarification response

8. RAG architecture

RAG is not only for documents. In this product it becomes the grounding system for user history, CV evidence, career frameworks, job market evidence, learning resources, and prior roadmap decisions.

Indexing pipeline

SourceChunking strategyMetadata
CV / ResumeSection-aware chunks: experience, skills, education, projectsuser_id, doc_id, sensitivity, extraction_confidence
Job descriptionsRequirement-aware chunks: responsibilities, must-have skills, nice-to-have skillsrole, company, region, posting_date, source_url
Roadmap artifactsPhase, milestone, task, habit, weekly reviewplan_id, status, completion, created_at
Learning resourcesCourse/module/book-level chunkslevel, cost, duration, provider, skill_tags
Market signalsAggregated evidence cards, not raw noisesource_quality, freshness, geography, confidence

Retrieval pipeline

  • Query rewriting: convert user goal into role, region, seniority, skill, and constraint queries.
  • Hybrid search: use keyword + vector retrieval to avoid missing exact tool names or job titles.
  • Re-ranking: rank by relevance, freshness, source quality, and user-fit.
  • Evidence cards: return compact facts with source, timestamp, confidence, and permissible usage.
  • Citation map: every generated recommendation points back to one or more evidence card IDs.

Evidence card schema

class EvidenceCard(BaseModel):
    id: UUID
    user_id: UUID | None
    source_type: Literal["user_cv", "job_posting", "course", "market_report", "roadmap_history", "manual_input"]
    source_ref: str
    claim: str
    extracted_facts: list[str]
    confidence: float
    freshness_date: date | None
    sensitivity: Literal["public", "user_private", "restricted"]
    allowed_usage: list[Literal["reasoning", "display", "export", "tool_input"]]

9. MCP tool layer

MCP should be treated as a controlled tool gateway, not a free-for-all plugin system. Tools are registered, permissioned, audited, and called only through policy checks.

Read tools

job.searchgithub.trendscourse.searchsalary.lookup

Allowed during evidence retrieval when the user has accepted data-source terms.

Write tools

calendar.createcrm.add_contacttask.createdraft_message

Require explicit user approval. The system must show the proposed action before execution.

Restricted tools

linkedin.connectemail.sendapplication.submit

Disabled by default. Enable only with scoped OAuth and explicit confirmation per action.

Tool call governance

ControlPurposeImplementation
Consent scopePrevents unexpected access to external accounts.Store user-granted scopes per integration and check before tool call.
Purpose bindingPrevents tools being used outside the active workflow.Each call includes run_id, agent_id, purpose, and policy decision.
Rate limitsProtects APIs and user accounts.Redis counters per user, tool, and integration.
Data minimisationLimits sensitive data exposure.Tool inputs receive only needed fields, not full user profile dumps.
AuditabilityMakes AI actions reviewable.Persist request hash, response hash, timestamp, status, and user-visible summary.

10. Agent-to-agent communication

Agents communicate through a shared run state and typed messages. This avoids brittle prompt chaining and allows each output to be inspected, retried, validated, or replaced.

Message envelope

{
  "message_id": "uuid",
  "run_id": "uuid",
  "from_agent": "profile_agent",
  "to_agent": "gap_analysis_agent",
  "message_type": "PROFILE_EXTRACTED",
  "payload_schema": "ProfileSnapshotV1",
  "payload": {},
  "evidence_refs": ["ev_123", "ev_456"],
  "confidence": 0.91,
  "created_at": "2026-04-29T10:30:00Z"
}

Shared memory boundaries

  • Short-term run memory: current workflow state and intermediate artifacts.
  • User memory: profile, goals, preferences, active plans, completed milestones.
  • Evidence memory: retrieved source cards with confidence and freshness.
  • Tool memory: MCP calls and returned structured data.
  • Decision memory: why a roadmap item was created, changed, rejected, or approved.

11. Hallucination prevention and realism controls

The system must make it difficult for unsupported claims to survive. Validation is done before the user sees a final roadmap and again before persistence.

Guardrail

No source, no claim

Company hiring claims, salary ranges, trending skills, visa guidance, certification value, course metadata, and local event recommendations must reference evidence cards or be labelled as assumptions.

Guardrail

Timeline feasibility scoring

The validator checks milestone volume against weekly hours, current skill level, target difficulty, and user constraints. Unrealistic plans are returned for repair.

Guardrail

Contradiction detection

The validator compares user statements, CV evidence, prior preferences, and generated plan. Contradictions trigger clarification or a visible warning.

Guardrail

Freshness policy

Market-sensitive data must include retrieved_at and source_date. Stale market signals are downgraded or excluded from high-impact recommendations.

Guardrail

Uncertainty display

Recommendations include confidence labels: high, medium, low. Low-confidence recommendations require explanation and optional user confirmation.

Guardrail

Schema validation

All LLM output is parsed into Pydantic schemas. Invalid output is rejected, repaired, and logged. Free-form text is never directly persisted as roadmap state.

Validation checklist

CheckPass conditionFailure action
Evidence coverageEvery major roadmap recommendation has evidence or a declared assumption.Ask RAG/market agent for more evidence or mark as assumption.
Personal fitPlan respects time budget, learning style, location, budget, and constraints.Regenerate constrained section.
Role alignmentMilestones map to target-role skill graph and gap report.Remove or re-rank irrelevant milestones.
MeasurabilityEach phase has concrete success metrics and evidence artifacts.Add metrics or reject phase.
Safety and biasAdvice avoids protected-attribute assumptions and does not limit ambition unfairly.Escalate to responsible-AI rewrite.

14. Infrastructure and deployment architecture

Roadmap generation can be slow because it may parse documents, retrieve evidence, call tools, and run multiple LLM steps. Use asynchronous workers instead of blocking HTTP requests.

FastAPI app

Handles auth, REST APIs, streaming endpoints, validation, and orchestration job creation. Keeps request latency low.

Agent workers

Consume jobs from a queue and run orchestration workflows. Workers can be scaled independently from the API.

Scheduler

Runs weekly review prompts, market refresh jobs, roadmap adaptation scans, and stale-evidence cleanup.

Queue

Use Celery/RQ/Arq with Redis or a production broker. Store job status in PostgreSQL for reliability.

Storage

PostgreSQL for state, object store for documents, vector store for embeddings, Redis for cache/rate limiting/session data.

Kubernetes-ready

Separate deployments for API, worker, scheduler, MCP gateway, embedding service, and observability stack.

15. Security, privacy, and responsible AI

The system handles CVs, career goals, progress history, and connected accounts. Security must be designed into every layer, especially agent tools and generated decisions.

Security controls

  • JWT access tokens, refresh tokens in HttpOnly Secure SameSite cookies.
  • Per-user row-level isolation in repositories and service methods.
  • Encryption at rest for uploaded documents and sensitive fields.
  • Tool-call audit logs for MCP actions.
  • Rate limiting on auth, chat, generation, and tool endpoints.
  • Secrets managed outside code through environment or secret manager.

Responsible AI controls

  • Do not infer protected attributes or reduce ambition based on them.
  • Expose uncertainty and assumptions to users.
  • Allow users to delete documents and derived analyses.
  • Require consent for external account connections.
  • Keep a human approval gate for write actions and critical plan changes.
  • Provide explanations for important recommendations.

19. Final backend blueprint

The production backend should combine your existing product modules with a new agentic AI core. The key decision is to treat AI outputs as draft artifacts that need evidence, validation, and user approval before they become persistent product state.

What the system generates

  • Career diagnosis and readiness score.
  • Skill gap report with evidence.
  • Multi-phase roadmap.
  • Monthly milestones and weekly execution plan.
  • Portfolio/project recommendations.
  • Learning resources and reading plan.
  • Networking and opportunity strategy.
  • Progress metrics and review cadence.

What the system must never do

  • Invent job-market facts or company requirements.
  • Assume personal constraints that the user did not provide.
  • Present uncertain career advice as guaranteed truth.
  • Use external accounts without scoped consent.
  • Send messages, apply to jobs, or modify calendars without approval.
  • Persist unvalidated LLM free text as source-of-truth state.
